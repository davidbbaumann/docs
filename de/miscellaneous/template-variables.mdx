Variablen sind Platzhalter, die bei der Nutzung von Vorlagen oder beim Ausführen eines KI-Prompts automatisch durch echte Werte aus Datensätzen ersetzt werden. Sie folgen immer dem Muster `{{namespace.feld}}` und können optional mit einem Fallback versehen werden: `{{candidate.first_name|"Kandidat:in"}}`.

<Tip>
  Über den Button **„Variablen"** im Editor (E-Mail, Dokumentenvorlage, Prompt) können alle verfügbaren Variablen per Klick eingefügt werden – inklusive Ihrer benutzerdefinierten Attribute.
</Tip>

## Syntax

| Beispiel | Beschreibung |
| --- | --- |
| `{{candidate.first_name}}` | Standard-Platzhalter |
| `{{candidate.first_name|"Kandidat:in"}}` | Fallback, falls das Feld leer ist |
| `{{application.emails}}` | Listenfeld – Werte werden mit Komma verkettet |

## Bewerbung

Daten aus einer eingegangenen Bewerbung.

| Variable | Referenz | Beschreibung |
| --- | --- | --- |
| Bewerbungs-ID | `{{application.id}}` | Eindeutige Kennung der Bewerbung |
| Vorname | `{{application.first_name}}` | Vorname der bewerbenden Person |
| Nachname | `{{application.last_name}}` | Nachname der bewerbenden Person |
| Vollständiger Name | `{{application.full_name}}` | Vor- und Nachname |
| E-Mail-Adresse(n) | `{{application.emails}}` | E-Mail-Adressen der bewerbenden Person |
| Telefonnummer(n) | `{{application.phones}}` | Telefonnummern der bewerbenden Person |
| Quelle | `{{application.source}}` | Herkunft der Bewerbung (z. B. LinkedIn) |
| Bewerbungsdatum | `{{application.created_at}}` | Eingangsdatum der Bewerbung |
| Bewerbungsstatus | `{{application.workflow_step_name}}` | Aktueller Workflow-Schritt |
| Consent-URL | `{{application.consent_url}}` | Persönlicher Link zur Verwaltung der Einwilligung |

## Benutzerdefinierte Attribute als Variablen

Alle **benutzerdefinierten Attribute** (Custom Attributes), die Sie unter **Einstellungen → Attribute** anlegen, stehen automatisch als Variablen zur Verfügung. Es ist keine zusätzliche Konfiguration notwendig.

### Aufbau der Referenz

Die Referenz setzt sich aus dem Entitätstyp (Namespace) und dem **API-Namen** des Attributs zusammen:

```text
{{<namespace>.<api_name>}}
```

Unterstützte Namespaces für Custom Attributes:

| Entität | Namespace |
| --- | --- |
| Bewerbung | `application` |
| Kandidat:in | `candidate` |
| Kontakt | `contact` |
| Unternehmen | `account` |
| Projekt | `project` |

### Beispiel

Sie haben für Kandidat:innen ein Custom Attribute mit dem Label **„Gewünschtes Gehalt"** und dem API-Namen `desired_salary` definiert. Die Variable lautet dann:

```text
{{candidate.desired_salary}}
```

<Tip>
  Den exakten API-Namen finden Sie in **Einstellungen → Attribute** unterhalb des Labels. Der API-Name wird beim Anlegen automatisch aus dem Label generiert (z. B. Label „Gewünschtes Gehalt" → API-Name `desired_salary`).
</Tip>

### Verhalten pro Attributtyp

| Attributtyp | Ausgabeverhalten |
| --- | --- |
| Text | Wert wird als Text eingefügt |
| Zahl | Wert wird als Zahl eingefügt |
| Datum | Wird gemäß Spracheinstellung formatiert |
| Ja/Nein | Wird zu „Ja" / „Nein" aufgelöst |
| Auswahl (Select) | Das **Label** der ausgewählten Option wird eingefügt (nicht der interne API-Name) |
| Mehrfachauswahl (Multiselect) | Alle ausgewählten Labels werden per Komma verkettet |

### Fallback verwenden

Auch bei Custom Attributes können Sie einen Fallback definieren, falls das Feld leer ist:

```text
{{candidate.desired_salary|"nicht angegeben"}}
```

---

<Note>
  Fehlen die Daten zum Zeitpunkt der Auflösung (z. B. weil ein Kandidat noch keine Telefonnummer hinterlegt hat), wird – falls definiert – der Fallback verwendet, ansonsten bleibt das Feld leer.
</Note>
